# JRuby CI Improvement - Research Notes

## TODO 1: Remove `:except_jruby` from elasticgraph-local specs

### Problem
Two acceptance tests in `elasticgraph-local/spec/acceptance/rake_tasks_spec.rb` (lines 23, 59) are tagged `:except_jruby`. They pass `--daemonize --pid <file>` to rackup, which uses `Process.daemon` (fork-based) internally. JRuby lacks `fork`.

### Rejected Approach (PR #1037, commit 2d96f249)
Modified `rake_tasks.rb` to detect `--daemonize` in rackup_args and use `Process.spawn` instead. Owner rejected this because it complicates the production implementation with JRuby-specific branching.

### Requirement
- Implementation stays simple: no branching on `--daemonize`
- JRuby workaround isolated to tests only
- A seam that tests hook into is acceptable

### Key Files
- `elasticgraph-local/lib/elastic_graph/local/rake_tasks.rb` line 486: the `sh` rackup call
- `elasticgraph-local/spec/acceptance/rake_tasks_spec.rb`: the two tagged specs
- `elasticgraph-local/lib/elastic_graph/local/docker_runner.rb`: existing `Process.spawn` + `Process.detach` pattern
- `spec_support/lib/elastic_graph/spec_support/rake_task.rb`: `RakeTaskSupport#run_rake` base implementation

### Proposed Seam
Extract `sh command` on line 486 into a `run_rackup(command)` private method. Tests override it at instance level on JRuby using `define_singleton_method` or `def t.run_rackup(...)`.

### How Tests Work
- `run_rake` in spec overrides `RakeTaskSupport#run_rake` via `super`
- Base creates Rake::Application, yields for task definition, then runs app
- Spec creates `RakeTasks.new(...)` in the yield block
- Task block captures `self` = RakeTasks instance
- Method lookup at task execution time finds singleton overrides

### Pattern Reference: DockerRunner
`docker_runner.rb` uses `Process.spawn` + `IO.pipe` + `Process.detach` for backgrounding. The JRuby test override can follow a similar pattern (simpler though, since we don't need log monitoring).

- Single CI job: `run_specs_for_jruby` takes 50+ minutes
- Runs `script/run_specs --format documentation` (all gems except elasticgraph-local)

### CI Structure
- CI yaml generated by `script/update_ci_yaml` (only updates datastore versions)
- JRuby entry is in the `include` section of the matrix
- `run` step: `script/ci_parts/${{ matrix.build_part }} ${{ matrix.datastore }} 10`
- `setup_env` handles datastore boot, exports gem lists

### Proposed Split (3 parts, targeting ~17 min each)
- **Part 1**: elasticgraph-graphql (80 specs, ~30% of total)
- **Part 2**: elasticgraph-schema_definition + elasticgraph-indexer (83 specs, ~31%)
- **Part 3**: All remaining gems + elasticgraph-local (107 specs, ~39%)

### Implementation Approach
- Parameterize `run_specs_for_jruby` to accept a part number ($3)
- Add `build_part_args` to CI matrix for JRuby entries
- Modify CI `run` step to pass `${{ matrix.build_part_args }}`
- Each part passes spec dirs to `script/flatware_rspec` (falls back to rspec on JRuby)

### CI Matrix Change
Replace single JRuby entry with 3 entries, each with different `build_part_args` ("1", "2", "3"). `update_ci_yaml` auto-updates datastore versions for all include entries - no changes needed there.
